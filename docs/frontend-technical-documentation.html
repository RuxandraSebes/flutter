Frontend Technical Documentation

Project: Spital mobile app (Flutter)

Codebase: spital_app

1. Architecture Overview

The frontend is a Flutter mobile client using a classic layered approach:

  • Presentation layer: screen widgets in lib/screens.
  • Service layer: REST API clients in lib/services.
  • Model layer: user model in lib/models/user_model.dart.
  • Localization layer: i18n maps and locale provider in lib/i18n.

2. Bootstrap and Navigation Flow

  • main.dart wraps the app in LanguageProvider and starts AuthGate.
  • AuthGate checks token presence and cached user data via AuthService.
  • roleBasedHome(user) routes users by role:
    • global_admin - Global admin dashboard
    • hospital_admin - Hospital admin dashboard
    • doctor - Doctor dashboard
    • patient / companion - Home screen

3. Feature Modules

3.1 Authentication and Profile

  • login_screen.dart: login form and auth execution.
  • register_screen.dart: registration with role/CNP/hospital support.
  • email_verification_screen.dart: code verification after registration.
  • claim_account_screen.dart: profile completion for auto-created patient accounts.
  • auth_service.dart: login/register/verify/resend/logout/me/profile update.

3.2 Documents

  • home_screen.dart and doctor_screen.dart list/upload/delete documents based on role permissions.
  • pdf_viewer_screen.dart handles PDF display and paging.
  • document_service.dart uploads and queries documents via API.

3.3 Companion Linking

  • generate_access_code_screen.dart creates numeric code or email invite.
  • redeem_access_code_screen.dart redeems access code/invite token.
  • invite_token_screen.dart deep-link invitation acceptance flow.
  • my_companions_screen.dart self-service manage linked companions/patients.
  • access_code_service.dart encapsulates all companion-link endpoints.

3.4 Chat

  • chat_conversations_screen.dart conversation list.
  • chat_screen.dart message thread per patient.
  • chat_service.dart handles conversation and send/read operations.

3.5 Administration

  • global_admin_screen.dart: manage hospitals and all users.
  • hospital_admin_screen.dart: manage hospital users and companion links.
  • doctor_screen.dart: patient view, document upload, medical messaging.
  • admin_service.dart: hospital/user CRUD and link/unlink API calls.

4. Localization and Message Translation

  • Translation maps: en.dart, ro.dart, hu.dart, uk.dart, sk.dart.
  • AppLocalizations.get(key) returns translated text by locale.
  • backend_message_mapper.dart normalizes backend plain-text messages to i18n keys so UI never exposes raw API text.
  • Language selection is persisted through flutter_secure_storage via LanguageProvider.

5. API Integration Contract

ServicePurposeMain Endpoints
AuthServiceSession and profile/login, /register, /auth/verify-email, /logout, /me, /profile
DocumentServiceDocuments/documents (GET/POST), /documents/{id} (DELETE)
AccessCodeServiceCompanion linking/access-codes/*, /my-companions, /my-patients
ChatServiceChat/chat/conversations, /chat/messages
AdminServiceAdmin operations/users, /hospitals, /companions/link, /companions/unlink

6. Configuration and Environments

  • Backend URL: configured in AuthService.baseUrl.
  • Auth: token-based API auth (Sanctum), token stored in secure storage.
  • Network requirement: mobile device must reach backend host/IP in baseUrl.

7. Error Handling Strategy

  • Services return { success: true/false, ... } maps.
  • Frontend translates keys from service responses before showing UI text.
  • Fallback key for generic failures is error or feature-specific keys.

8. Security Notes

  • Auth token is stored with flutter_secure_storage.
  • Client enforces role-based screen routing, but backend is the final authority for permissions.
  • Do not hardcode production URLs or credentials in source for release builds.
