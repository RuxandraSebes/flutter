Backend Technical Documentation

Project: Spital backend API (Laravel)

Codebase: spital-backend

1. Technology Stack

  • PHP + Laravel framework
  • Eloquent ORM and Laravel migrations
  • Laravel Sanctum for token authentication
  • Role-based middleware authorization
  • REST JSON API consumed by Flutter mobile app

2. High-Level Architecture

  • Routes: routes/api.php exposes all public and protected endpoints.
  • Controllers: Auth, Documents, Access Codes, Admin, Chat, Profile, PDF Ingestion.
  • Models: User, Document, Hospital, CompanionAccessCode, ChatMessage, IngestLog.
  • Middleware: CheckRole enforces role constraints on route groups.

3. Authentication and Authorization

3.1 Authentication

  • Login creates Sanctum tokens.
  • Protected routes are grouped under auth:sanctum.
  • Current user endpoint: GET /api/me.

3.2 Role System

Role constants are defined in App\Models\User:

  • global_admin
  • hospital_admin
  • doctor
  • patient
  • companion

CheckRole middleware validates:

  • authenticated user exists
  • user account is active (is_active)
  • user role is one of the accepted roles for the endpoint

4. API Surface (Summary)

AreaEndpointsDescription
Auth POST /register, POST /login, POST /auth/verify-email, POST /auth/resend-verification, POST /logout, GET /me User registration, login, email verification, session lifecycle.
Hospitals GET /hospitals/public, GET/POST/PUT/DELETE /hospitals (admin) Public hospital list for registration and admin CRUD for hospital entities.
Documents GET /documents, POST /documents, DELETE /documents/{id} Upload, list, delete medical PDFs with role-aware access logic.
Companion Access POST /access-codes/generate, POST /access-codes/redeem, POST /access-codes/invite, POST /access-codes/invite/redeem Numeric and email invite linking between patients and companions.
Self-service links GET/DELETE /my-companions, GET/DELETE /my-patients Patient and companion can view/remove existing links.
Chat GET /chat/conversations, GET /chat/messages, POST /chat/messages Role-aware messaging between doctor side and patient side.
Admin users GET /users, POST /users, PUT /users/{id}, DELETE /users/{id}, POST /companions/link, POST /companions/unlink User management and patient-companion link management.
Ingestion POST /ingest/pdf, POST /ingest/poll, GET /ingest/status Automated ingest pipeline for incoming PDFs and operational status visibility.

5. Data Model (Core Tables)

  • users: identity, role, CNP, hospital, specialization, active flag, email verification fields.
  • hospitals: master data for hospital entities.
  • documents: uploaded medical files linked to user/patient owner.
  • patient_companions: many-to-many patient/companion relationship with metadata.
  • companion_access_codes: temporary numeric codes / invite tokens with TTL and used flag.
  • chat_messages: conversation messages including sender role marker.
  • ingest_logs: ingestion operational events and outcomes.

6. Business Rules

  • Only active users are authorized to use protected endpoints.
  • Patient-companion linking is temporary-token based (numeric code or email invite).
  • Document access is role constrained: users see only what they are authorized to read.
  • Admin actions are split:
    • Global admin: full hospital CRUD + full user scope.
    • Hospital admin: user write scope and companion link management.

7. Response and Error Strategy

  • API returns JSON with success payloads or message strings for error/info.
  • Frontend maps backend messages to localization keys (backend_message_mapper.dart).
  • Status codes reflect semantics: 200/201 success, 401 unauthenticated, 403 forbidden, 422 validation/semantic issues.

8. Environment and Deployment Basics

composer install
cp .env.example .env
php artisan key:generate
php artisan migrate
php artisan serve --host=0.0.0.0 --port=8000

For production, use proper web server (Nginx/Apache), HTTPS, queue workers if used, and secured environment variables.

9. Security Recommendations

  • Restrict CORS origins in production.
  • Rate-limit login and token redemption endpoints.
  • Store API keys and mail credentials only in environment variables.
  • Add audit logging for admin write actions if compliance requires traceability.

10. Operational Notes

  • /ingest/status endpoint is useful for support and monitoring pipeline health.
  • Database migrations include progressive features (email verification, chat, ingest logs, companion token invites).
  • Keep frontend AuthService.baseUrl aligned with actual backend environment.
